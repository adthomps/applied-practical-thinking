# Validation Results Runbook

This runbook explains how to run checks, where outputs are written, and how to interpret and action the results.

## Scope

This guide covers:

- Canonical doctrine validation in `apt-principles`
- Workspace project-profile sweep and portfolio dashboard outputs
- Repo-level lint, typecheck, and test verification for target repos

## Prerequisites

- Node.js and npm installed
- pnpm installed for repos that use pnpm workspaces
- Workspace layout where `apt-principles` is a sibling of target repos

## Command Quick Start

Run from `apt-principles`:

```bash
npm run run-all-checks
npm run validate
npm run sweep:project-profiles
```

Use `npm run run-all-checks` for the default operator flow when you want one command that runs canonical checks plus selected repo quality gates and then prints a compact summary.

Run repo quality gates in each target repo root:

```bash
pnpm lint
pnpm typecheck
pnpm test
```

## Recommended Execution Order

Fast path:

```bash
cd apt-principles
npm run run-all-checks
```

Expanded manual path:

1. Validate canonical doctrine and references:

```bash
cd apt-principles
npm run validate
```

2. Run the workspace sweep to produce consolidated and per-repo outputs:

```bash
npm run sweep:project-profiles
```

3. Run repo-local engineering checks for any repo in scope:

```bash
cd ../apt-novel-reviewer
pnpm lint
pnpm typecheck
pnpm test

cd ../crt-world
pnpm lint
pnpm typecheck
pnpm test
```

4. Read the generated markdown dashboard and action queue first, then inspect JSON for automation.

## Canonical Validation

- Validation command: `npm run validate`
- Primary output: terminal summary and exit code
- Success signal: exit code `0`
- Failure signal: non-zero exit code with structural/content errors in output

## Sweep Outputs

Generated by `npm run sweep:project-profiles`:

- Portfolio markdown dashboard:
  - `apt-principles/reports/project-profile-validation-sweep-YYYY-MM-DD.md`
- Portfolio machine summary:
  - `apt-principles/reports/project-profile-validation-sweep-YYYY-MM-DD.json`
- Repo machine summaries:
  - `<repo>/docs/apt/reports/static/project-profile-validation-sweep-YYYY-MM-DD.json`

## How To Read The Markdown Dashboard

Open the latest file under `apt-principles/reports/` and review sections in this order:

1. `Summary`
   - Confirms scan coverage and pass/fail/skip counts.
2. `Principle Coverage (Rubric)`
   - Shows counts of `Pass`, `Partial`, `Gap`, and `Not Applicable` by APT layer.
3. `Findings by Severity`
   - Prioritize `Blocking` and `Major` first.
4. `Findings by APT Layer`
   - Reveals concentration areas (for example Security, Quality, Design).
5. `Repo Status` and `Top Findings`
   - Quick repo-by-repo triage context.
6. `Portfolio Action Queue`
   - Top suggested next actions across repos.

## How To Read The JSON Outputs

Each JSON payload includes:

- `status` and `errors` for validation state
- `findings` and `top_findings` for evidence-backed issue summary
- `rubric_scores` for principle-level score extraction
- `top_actions` for next-step queue items
- `workspace_summary` for portfolio rollups and grouping counts

Use JSON when you need:

- machine processing in scripts or CI annotations
- downstream dashboards
- issue generation and triage automation

## Triage Direction

Use this default triage order:

1. All `Blocking` findings
2. All `Major` findings
3. Layers with highest finding concentration
4. Rubric layers with high `Partial` and any `Gap` counts
5. Portfolio action queue items that unblock multiple repos

## CI Usage Pattern

In each project CI workflow:

1. Check out the target repo.
2. Check out `apt-principles` as a sibling path (for shared scripts).
3. Run profile validation if `docs/apt/references/project-profile.json` exists.
4. Run repo-local lint/typecheck/test commands.

Expected CI behavior:

- Missing local profile file can be treated as a skip if adoption is not complete.
- Invalid profile shape must fail the validation step.
- Lint/typecheck/test failures must fail the workflow.

## Common Failure Modes

- Missing `project-profile.json` in a repo expected to participate in the sweep
- Invalid `adoption_mode`, `maturity`, or malformed `showcase` object
- Audit markdown missing parsable `Findings` or `Rubric` sections
- Repo scripts present but not wired at package-level for lint/typecheck/test

## Recommended Cadence

- Run `npm run validate` whenever doctrine/references/templates are edited.
- Run `npm run sweep:project-profiles` after updates to project profiles or audit reports.
- Run repo-local lint/typecheck/test before merge and in CI on every PR.

## Ownership

- Canonical doctrine, sweep logic, and consolidated reports: `apt-principles`
- Repo-specific evidence and static outputs: each target repo under `docs/apt/reports/static/`
